Negação e dif/2 em Prolog

May 6, 2021

Uma das dificuldades ao programar em Prolog é lidar com negação. Os fatos de um banco de dados definem um certo espaço onde essas coisas são verdade. As cláusulas ampliam esse espaço, extrapolando a partir dos fatos. Provar uma negativa, do outro lado, exigiria saber o universo inteiro, para então retirar dele os fatos conhecidos!

Por exemplo, vamos listar alguns membros da família real britânica, anotando quem são seus pais com a relação parent(PaiOuMãe, FilhoOuFilha):

parent(george, elizabeth).
parent(george, margaret).

parent(elizabeth, charles).
parent(philip, charles).

parent(charles, william).
parent(diana, william).
parent(charles, harry).
parent(diana, harry).

Eu fiz essa lista de cabeça, com conhecimentos adquiridos pela série “The Crown”. Não está listada, por exemplo, a mãe da Rainha Elizabeth.

Podemos agora ampliar nosso universo de conhecimento descrevendo uma relação derivada a partir de parent/2. Por exemplo, vamos descrever um avô/avó como o genitor de genitor de alguém.

grandparent(Gran, Child) :-
    parent(Gran, Parent),
    parent(Parent, Child).

Escrever esta regra expande o espaço contendo todas as verdades conhecidas. Podemos perguntar agora quais são todos os avôs e avós, como se tivéssemos escrito cada uma dessas relações por extenso.

?- grandparent(X, Y).
X = george, Y = charles ;
X = elizabeth, Y = william ;
X = elizabeth, Y = harry ;
X = philip, Y = william ;
X = philip, Y = harry ;
false.

Legal! Podemos também definir uma nova relação para descobrir quem é irmão de quem com sibling/2, onde se define que irmãos são duas pessoas que possuem pelo menos um genitor em comum.

sibling(Child1, Child2) :-
    parent(Parent, Child1),
    parent(Parent, Child2).

Agora podemos consultar pela irmã da Princesa Margaret, por ex-

?- sibling(margaret, X).
X = elizabeth ;
X = margaret.

Hmm, nossa definição de “irmão” está um pouco falha. Uma pessoa possui um pai em comum consigo mesma, então ela sempre vai aparecer na consulta. Podemos reescrever a cláusula para filtrar este caso:

sibling(Child1, Child2) :-
    Child1 \= Child2,
    parent(Parent, Child1),
    parent(Parent, Child2).

Agora podemos consultar novamente pela irmã da Princesa Margaret-

?- sibling(margaret, X).
false.

?- sibling(margaret, elizabeth).
true.

Essa não! Nós conseguimos consultar se “Elizabeth é irmã de Margaret?”, mas não conseguimos consultar “para quais X, X é irmã de Margaret?”. Isso torna nosso programa mais limitado, e a documentação de \= nos explica porquê:

@Term1 \= @Term2

Equivalente a \+ Term1 = Term2

Este predicado é logicamente correto se seus argumentos forem suficientemente instanciados. Em outros casos, como ?- X \= Y., o predicado falha mesmo existindo soluções. Isto se dá devido à natureza incompleta de \+/1.

Precisamos então reordenar os goals na cláusula, para garantir que \= seja usado com essas variáveis já instanciadas.

sibling(Child1, Child2) :-
    parent(Parent, Child1),
    parent(Parent, Child2),
    Child1 \= Child2.

Agora, podemos realizar a consulta em diferentes níveis de instanciação:

?- sibling(X, margaret).
X = elizabeth ;
false.

?- sibling(harry, X).
X = william ;
X = william ;
false.

Puxa, essa duplicação deixa nosso resultado muito feio. Isso acontece porque Harry é irmão de William por dois “caminhos”, com Parent=charles e Parent=diana. Seria interessante tentar retornar apenas uma vez. Podemos fazer um filtro posterior com setof/3, mas vamos explorar como garantir que emitimos se são irmãos apenas uma vez.

Para começar, precisamos considerar as possíveis combinações: Child1 e Child2 podem ter 1 ou 2 genitores. Se algum deles tiver apenas 1 genitor, então esse é o único caminho para serem irmãos. Se ambos possuírem 2 genitores, então devemos ser bem sucedidos apenas com um deles – digamos, o menor na ordem padrão dos termos.

sibling(Child1, Child2) :-
    % Child1 e Child2 compartilham um genitor Parent1.
    parent(Parent1, Child1),
    parent(Parent1, Child2),
    Child1 \= Child2,

    % Se Child1 e Child2 compartilharem um segundo genitor
    % Parent2 diferente de Parent1, suceda se tiver escolhido
    % primeiro o menor dos dois.
    % Caso essa condição falhe, suceda pois eles são irmãos
    % apenas via Parent1.
    ( parent(Parent2, Child1),
      Parent2 \= Parent1,
      parent(Parent2, Child2) ->
      Parent1 @< Parent2
    ; true
    ).

Agora, temos um resultado limpinho para qualquer consulta, às custas de um código mais complicado:

?- sibling(X, margaret).
X = elizabeth ;
false.

?- sibling(harry, X).
X = william ;
false.

Usando dif

Eu fui desonesto ao explicar esse caso para você, pois omiti que a própria documentação de \=/2 já contém uma dica de como fazer melhor:

Para fazer com que seu programa funcione corretamente quando os argumentos não estiverem suficientemente instanciados, use dif/2.

Historicamente, dif/2 já estava presente desde o primeiro Prolog, e consiste em uma função que verifica se dois termos são diferentes e impede no futuro que alguma instanciação faça com que eles sejam iguais. Acompanhe os seguintes exemplos:

% X não está instanciado, e o resultado é apenas a
% restrição X ≠ 2
?- dif(X, 2).
dif(X, 2).

% X pode ser instanciado com 1.
?- dif(X, 2), X = 1.
X = 1.

% Tentar instanciar X com 2 falha.
?- dif(X, 2), X = 2.
false.

% Tentar instanciar X com 2 falha, mesmo se estiver
% dentro de um termo composto.
?- dif(f(X), f(2)), X = 2.
false.

% Neste caso, X ≠ Y, e X é instanciado com 2.
% Y não é instanciado, e o resultado inclui a
% restrição Y ≠ 2.
?- dif(f(X), f(Y)), X = 2.
X = 2,
dif(Y, 2).

Esse recurso permite tratar a negação diretamente, como uma propriedade a ser satisfeita. Mesmo que não possamos enumerar todo o universo e retirar apenas os valores negados, podemos ainda raciocinar sobre as propriedades do espaço da solução. No último exemplo acima, sabemos que Y ≠ 2 é uma condição suficiente para a solução ser verdadeira. Se quiséssemos obter todas as soluções, teríamos que enumerar via backtracking todos os inteiros que satisfazem isso!

Usando dif/2, nosso código fica bem mais simples, e não precisamos nos preocupar com a ordem dos termos na cláusula.

sibling(Child1, Child2) :-
    % Child1 e Child2 são diferentes e
    % compartilham um genitor
    dif(Child1, Child2),
    parent(Parent1, Child1),
    parent(Parent1, Child2),

    % Se ambas compartilham um genitor diferente
    % de Parent1, falhe se não tiver escolhido
    % primeiro o menor deles.
    dif(Parent1, Parent2),
    ( parent(Parent2, Child1),
      parent(Parent2, Child2) ->
      Parent1 @< Parent2
    ; true
    ).

Como dif/2 funciona?

Em SWI-Prolog, dif/2 é implementado utilizando variáveis com atributos (que daqui em diante vou chamar só de attrvars). Attrvars são variáveis desvinculadas (unbound) associadas a atributos, que por sua vez são termos lógicos usuais com um twist: durante a unificação dessa variável, uma função associada ao atributo é invocada, que pode fazer a unificação falhar ou até modificar o valor que está sendo vinculado.

O objetivo de attrvars é, em suma, permitir estender o mecanismo de unificação. É uma funcionalidade de baixo nível, muito difícil de utilizar corretamente, mas também muito poderosa para a solução de sistemas de restrições, segundo Marcus Triska. Attrvars são utilizadas para implementar não só dif/2 em SWI-Prolog como também freeze/2, que permite atrasar uma execução até que uma variável seja vinculada; e todas as bibliotecas de programação de restrições lógicas (CLP, de Constraint Logic Programming).

Como estou implementando minha própria engine lógica, resolvi implementar attrvars e então tentar implementar dif/2. Este post é basicamente um estudo (prolixo) da implementação de SWI-Prolog.

Em SWI-Prolog, um atributo está sempre associado a um módulo. Quando uma variável contendo um atributo é vinculada, a função attr_unify_hook(+Atributo, +Valor) do módulo é invocada com o atributo e o valor vinculado. No caso de dif/2, o módulo se chama dif; com isso vamos investigar o que acontece com diferentes chamadas à função utilizando a primitiva get_attr(+Var, +Módulo, +Valor).

?- dif(X, 1), get_attr(X, dif, Attr).
Attr = vardif([_5678-1], []),
dif(X, 1).

A primeira resposta é o que estamos interessados, e a segunda é apenas a repetição dos termos residuais, ou restrições não satisfeitas. Daqui em diante vou omitir estes termos.

Diferentes chamadas a dif/2 geram termos adicionais no atributo de X: pares de variáveis desvinculadas com o valor de diferença.

?- dif(X, 1), dif(X, 2), dif(X, 10), get_attr(X, dif, Attr).
Attr = vardif([_7034-10, _7046-2, _7058-1], []), ...

Estudando o código do módulo dif vemos que uma chamada a dif/2 invoca unifiable/3, um predicado nativo que testa se dois termos são unificáveis, e quais substituições de variáveis satisfazem essa equivalência. De certa maneira, ela concretiza a unificação ao mostrar como ela seria feita, ao invés de fazê-la diretamente.

?- unifiable(f(A, 2), f(1, B), Unifier).
Unifier=[B=2, A=1].

?- f(A, 2) = f(1, B).
A=1, B=2.

Ao ser invocado, dif/2 testa a unificação de seus dois termos, e lista as substituições que os tornariam equivalentes. Se alguma das variáveis for vinculada com um desses valores, a unificação falha.

O resultado de unifiable/3 não é armazenado diretamente como atributo da variável, mas sim como um atributo de uma variável OrNode, que essa sim é referenciada pelo atributo da variável.

?- dif(X, 1), get_attr(X, dif, vardif([Or-1], [])), get_attr(Or, dif, AttrOr).
AttrOr = node(1, [X=1]), ...

Qual é a função desta indireção? O modelo é similar ao de um grafo, onde cada variável seria um vértice e o OrNode seria a aresta que os conecta. Todos os vértices participantes de uma mesma chamada ao dif/2 apontam para a mesma estrutura.

?- dif(f(A,2), f(1,B)),
   get_attr(A, dif, AttrA),
   get_attr(B, dif, AttrB),
   AttrA = vardif([Or-1], []),
   get_attr(Or, dif, AttrOr).
AttrA = vardif([Or-1], []), % AttrA e AttrB apontam
AttrB = vardif([Or-2], []), % para Or
AttrOr = node(2, [B=2, A=1]), ...

Os exemplos anteriores mostram que essas arestas são armazenadas no primeiro termo de vardif(V1, V2). Para que serve o segundo termo, então? A resposta aparece quando criamos uma relação de diferença entre duas variáveis X e Y:

?- dif(X, Y),
   get_attr(X, dif, AttrX),
   get_attr(Y, dif, AttrY),
   AttrX = vardif([Or-Y], []),
   get_attr(Or, dif, AttrOr).
AttrX = vardif([Or-Y], []), % Or-Y no 1º termo
AttrY = vardif([], [Or-X]), % Or-X no 2º termo
AttrOr = node(1, [X=Y]), ...

Assim, vemos que as arestas possuem uma direção, dependendo se a variável aparece do lado esquerdo ou direito de uma substituição. As arestas incidentes (isso é, de entrada) ficam em V2, e as arestas adjacentes (de saída) ficam em V1.

Aquilo que chamamos de “aresta” é uma variável que contém um atributo node(N, Pairs). O objetivo de usar uma attrvar ao invés de simplesmente associar um termo é utilizar-se da estrutura de forma mutável. Uma vez que uma variável referenciada pela aresta é vinculada, a estrutura é atualizada para 1) indicar que restrições foram satisfeitas; ou 2) remover um par agora equivalente da base, potencialmente falhando a unificação.

(Note que esta “mutabilidade” é diferente em Prolog: uma “alteração” de atributo registra o valor anterior, e é desfeita no backtrack.)

Vamos ver o que acontece quando vinculamos uma das variáveis acima.

?- dif(X, Y),
   get_attr(X, dif, AttrX),
   AttrX = vardif([Or-Y], []),
   Y = 1,
   get_attr(Or, dif, AttrOr).
AttrX = vardif([Or-1], []),   % Era Or-Y
AttrOr = node(1, [X=1]), ...  % Era X=Y

As referências a Y foram substituídas, mas o número que é o 1º argumento de node representa? Aparentemente, ele começa como uma contagem de todos os pares da aresta, e é decrementado quando um dos pares torna-se verdadeiro.

?- dif(f(A, 2), f(1, B)),
   get_attr(A, dif, vardif([Or-1], [])),
   get_attr(Or, dif, AttrOr1),
   format("AttrOr1: ~w~n", AttrOr1),
   B = 2,
   get_attr(Or, dif, AttrOr2),
   format("AttrOr2: ~w~n", AttrOr2).
AttrOr1: node(2,[_1824=2,_1868=1])
AttrOr2: node(1,[2=2,_1868=1])
...

No exemplo acima foi necessário utilizar format/2 para escrever imediatamente o estado do termo naquele ponto, pois dado que o atributo é mutável, tanto AttrOr1 quanto AttrOr2 apontam para o mesmo endereço.

De início eu fiquei confuso com este resultado. Se f(A,2) ≠ f(1,B), isto não implica que A ≠ 1 e B ≠ 2? A resposta é quase isso: a implicação é que A ≠ 1 ou B ≠ 2. A relação ainda é satisfeita com {A=1, B=20}, ou {A=10, B=2}. O que não satisfaz é ter ao mesmo tempo {A=1, B=2}. Entendeu agora porque eles se chamam OrNodes?

O uso do contador é uma otimização; poderíamos simplesmente manter a lista de restrições e remover os elementos que se mostram iguais. Contudo, isso exigiria buscar o elemento modificado na lista, e então remover esse nó do meio. Uma opção mais simples é notar que não precisamos saber qual elemento da aresta foi modificado, apenas que um deles foi vinculado a um valor “proibido”. Se o número de elementos diferentes cair a zero, então a unificação toda falha.

Se é assim, então qual a necessidade de se manter toda a lista? Ela ainda é útil para imprimir quais são as restrições ainda ativas na variável desvinculada. É assim que são obtidos os termos residuais, aqueles impressos ao final de uma consulta, se uma variável atribuída ainda está desvinculada.

Resumo

Quando se chama dif(A, B)…

  1. dif(A, B) invoca unifiable(A, B, Pairs), onde Pairs é uma lista de termos da forma X=Y, representando as substituições que unificam A e B;
  2. Cria uma variável Or com atributo node(N, Pairs), onde N é o comprimento de Pairs;
  3. Para as variáveis do lado esquerdo da substituição X=Y , adiciona a aresta Or-Y à lista de arestas adjacentes no atributo vardif(Adj, Inc);
  4. Para as variáveis do lado direito da substituição X=Y, adiciona a aresta Or-X à lista de arestas incidentes no atributo vardif(Adj, Inc).

Quando se tenta unificar X=V…

  1. Se X tem um atributo vardif(Adj, Inc), então…
  2. Se V é uma variável, mescla todas as restrições em ambas e atribui a V;
  3. Se V não é uma variável, para cada aresta Or-Y em Adj e Inc…
  4. Se Y \== V, adiciona em Or as restrições para garantir que Y ≠ V;
  5. Se Y == V, decrementa o contador do atributo node(Cnt, Pairs) de Or. Se o contador chegar a zero, a unificação falha.
