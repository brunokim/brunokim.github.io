Paradigmas de programação

As principais linguagens de programação em uso atualmente -- C++, Java, PHP, JavaScript, etc. -- seguem um paradigma procedural, com toques de orientação a objetos. Isso significa que o programador descreve os procedimentos que o computador deve executar para realizar uma tarefa. Para inverter uma lista, escrevemos que o computador deve executar n/2 vezes a troca de elementos entre o começo e o fim:

for (i = 0; i < n/2; i++) {
    let tmp = list[i];
    list[i] = list[n-i-1];
    list[n-i-1] = tmp;
}

Isso é relativamente próximo da linguagem procedural mais básica, o assembly utilizado para programar o processador. Assembly consiste em uma sequência de instruções simples, que são implementadas em hardware (ou próximo disso) pelo processador. Uma instrução como movl %esi, %r9d ("mova 32 bits do registro source index para o registro 9") necessita apenas de componentes eletrônicos básicos para ser implementada.

Este modelo, com suas origens desde a Máquina de Turing, não é o único meio de se pensar em computação. O próprio Turing mostrou que ele era equivalente ao modelo de Church, o Cálculo Lambda, que consiste apenas em expressões e chamadas de funções (ou: abstrações e aplicações).

true_ = (then_, else_) => then_();
false_ = (then_, else_) => else_();
if_ = (cond, then_, else_) => cond(then_, else_);
...
reverse = (list) => if_(
    equal(list, empty),
    () => empty,
    () => append(reverse(tail(list)), head(list)));

Este modelo deu origem ao paradigma funcional de programação, que prefere descrever programas como funções, onde as estruturas de dados são imutáveis e a operação de cada função depende apenas dos seus argumentos. Este paradigma ganhou mais atenção na última década como uma maneira mais robusta para implementar programas concorrentes e distribuídos -- ao limitar a mutabilidade, pouco importa se um pedaço de código será executado no processador A ou B, ou em uma máquina do outro lado do continente.

O paradigma de objetos (que o próprio Alan Kay gostaria de ter renomeado para passagem de mensagens) não surgiu de um modelo de programação universal, mas da área de simulações. Existe um outro paradigma, que, de fato, surgiu de um modelo alternativo: o paradigma lógico.

Programação lógica

A programação lógica surge da automação das cláusulas de Horn, um modelo genérico de programação. Estas cláusulas são da forma se \(p_1\) e \(p_2\) e \(p_3\) então \(u\).

Por exemplo, seja \(l\) uma lista, e \(a :: b\) signifique a concatenação das listas \(a\) e \(b\).

  • se \(l = \emptyset\) então \(\operatorname{reverse}(l) = \emptyset \)
  • se
    1. \(l = [h] :: l_1\); e
    2. \(\operatorname{reverse}(l_1) = r_1\); e
    3. \(r = r_1 :: [h]\); então
    4. \(\operatorname{reverse}(l) = r\)

Em outros termos, a reversão de uma lista vazia é a própria lista vazia; e a reversão \(r\) de uma lista \(l\) com primeiro elemento \(h\) e os restantes em \(l_1\) é a concatenação da reversão \(r_1\) de \(l_1\) com a lista \([h]\).

Estas cláusulas são normalmente expressas da forma \(u \leftarrow p_1 \wedge p_2 \wedge p_3\), que pode ser lido como "\(u\) é implicado por \(p_1\), \(p_2\) e \(p_3\)". A linguagem Prolog utiliza esta representação para cláusulas, usando :- para significar a setinha e vírgulas para significar a conjunção ("e").

reverse([], []).
reverse(L, R) :-
    concat([H], L1, L),
    concat(R1, [H], R),
    reverse(L1, R1).

Note que não escrevemos reverse([]) = [] ou L = concat([H], L1). Em Prolog puro não existem parâmetros de entrada ou saída de uma função, apenas relações entre termos. Com o programa acima, podemos fazer uma consulta tanto por R...

?- reverse([1, 2, 3, 4], R).
R = [4, 3, 2, 1].

...quanto por L...

?- reverse(L, [a, b, c, d]).
L = [d, c, b, a].

...ou por listas que são suas próprias inversas...

?- reverse(X, X).
X = [] ;
X = [_1] ;
X = [_1, _2, _1] ;
X = [_1, _2, _2, _1] ;
X = [_1, _2, _3, _2, _1]

...ou checar se uma consulta é verdade.

?- reverse([a, b, c], [b, c, a]).
false
